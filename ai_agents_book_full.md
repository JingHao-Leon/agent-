# 第1章：AI Agent概述

## 1.1 什么是AI Agent

### 1.1.1 Agent的定义

**AI Agent（人工智能代理）** 是一种能够自主理解目标、规划任务路径、执行复杂操作并根据反馈迭代优化的智能系统。与传统的**大语言模型（LLM）对话**不同，Agent不仅能"回答问题"，更能"完成任务"。

简单来说：

- **LLM对话** = 你说一句话，它回复一段话
- **AI Agent** = 你说一个目标，它自主完成一系列操作并汇报结果

### 1.1.2 传统LLM对话 vs AI Agent

| 对比维度 | 传统LLM对话 | AI Agent |
|---------|------------|----------|
| **交互模式** | 单轮问答 | 多轮自主规划 |
| **任务执行** | 仅生成文本 | 调用工具、执行动作 |
| **记忆能力** | 无状态（每次需重复上下文） | 持久记忆、跨会话连续 |
| **自主性** | 完全依赖人类指令 | 目标导向、自主决策 |
| **外部交互** | 无法访问外部系统 | 可操控API、数据库、文件系统 |
| **错误处理** | 一次性回答 | 自我反思、迭代修正 |
| **适用场景** | 问答、写作、翻译 | 自动化流程、数据分析、助手应用 |

### 1.1.3 AI Agent的核心特征

一个典型的AI Agent具备以下核心特征：

1. **自主规划（Autonomous Planning）**：将复杂任务分解为可执行的子任务
2. **工具使用（Tool Use）**：调用外部API、操作文件系统、查询数据库
3. **记忆管理（Memory Management）**：维护短期上下文和长期知识
4. **持续迭代（Self-Election）**：根据执行结果反馈调整下一步行动
5. **目标导向（Goal-Oriented）**：围绕最终目标而非单纯响应输入

## 1.2 Agent的四层演进

AI Agent的发展经历了四个层次的演进，从简单的问答交互到完全自主的行动系统。

### 1.2.1 Level 1：直接交互（Direct Interaction）

**特点**：

- 用户直接与LLM对话
- 纯文本输入/输出
- 无状态，每次交互独立
- 典型应用：ChatGPT对话

**局限性**：

- 无法执行实际动作
- 无法访问实时信息
- 上下文有限

### 1.2.2 Level 2：代理中介（Agent Mediation）

**新增能力**：

- Agent作为用户与LLM之间的中介
- 增加路由逻辑（判断意图）
- 可进行内容过滤和增强
- 典型应用：客服机器人

**增强点**：

```
用户输入 → 意图识别 → 路由选择 → LLM处理 → 响应增强 → 用户
```

### 1.2.3 Level 3：工具调用（Tool Augmentation）

**核心突破**：

- LLM可以调用外部工具
- Function Calling / Tool Use
- 访问API、搜索、文件操作
- 典型应用：GPTs、插件系统

**工作流程**：

```
1. 用户提出需求
2. LLM判断需要调用的工具
3. 执行工具获取结果
4. 将结果纳入上下文
5. 生成最终回答
```

### 1.2.4 Level 4：完全自主（Full Autonomy）

**最高形态**：

- Agent自主规划任务路径
- 动态决策执行顺序
- 自我评估与迭代修正
- 典型应用：AutoGPT、BabyAGI

**自主程度**：

| 能力维度 | Level 1 | Level 2 | Level 3 | Level 4 |
|---------|--------|--------|--------|--------|
| 任务分解 | 无 | 简单路由 | 静态分解 | 动态分解 |
| 工具调用 | 无 | 无 | 按需调用 | 自主选择 |
| 错误处理 | 无 | 简单反馈 | 预设重试 | 智能修正 |
| 自我评估 | 无 | 无 | 无 | 完整反思 |
| 多步协作 | 无 | 无 | 串行执行 | 并行/串行混合 |

## 1.3 Agent架构全景图

如前文的图1-1所示，AI Agent采用六层架构设计。每一层承担特定职责，层与层之间协同工作。

### 1.3.1 各层职责详解

**第一层：用户交互层**

用户交互层是Agent与用户之间的界面，负责：

- 接收用户指令（文本、语音、多模态）
- 格式化输出（文本、图表、操作结果）
- 管理会话状态（多轮对话连续性）
- 提供反馈机制（执行进度、结果确认）

**第二层：Agent核心层**

核心层定义Agent的"身份"和"大脑"：

- **Profile/Persona**：定义Agent的角色定位、性格特征、专业领域
- **Planning/Feedback**：任务规划引擎，负责将复杂任务分解为可执行步骤，并根据反馈调整计划

**第三层：推理评估层**

推理评估层是Agent的"思考能力"来源：

- **Reasoning**：逻辑推理引擎，处理复杂问题的思考过程
- **Evaluation**：结果评估模块，判断输出质量和任务完成度
- **Reflection**：自我反思机制，分析失败原因并生成改进策略

**第四层：记忆知识层**

记忆知识层赋予Agent"记忆"能力：

- **Memory（记忆）**：
  - 短期记忆：当前会话的上下文窗口
  - 长期记忆：跨会话积累的知识和偏好
- **Knowledge（知识）**：
  - 向量知识库：文档嵌入后的语义检索
  - 知识图谱：实体关系的结构化表示

**第五层：动作工具层**

动作工具层是Agent"动手做事"的能力：

- **基础工具**：文件读写、网络搜索、代码执行
- **API调用**：RESTful API、GraphQL、WebSocket
- **数据库**：SQL查询、向量检索、图数据库
- **自定义工具**：用户定义的业务逻辑接口

**第六层：LLM底座层**

LLM底座层提供基础语言智能：

- 文本理解与生成
- 意图识别与推理
- 上下文学习与指令跟随

### 1.4.1 Profile / Persona（角色定义）

Profile定义了Agent的身份和行为准则：

```yaml
# Agent Profile 示例
profile:
  name: "文档助手小Doc"
  type: "专业文档助手"
  expertise:
    - 技术文档撰写
    - 代码注释生成
    - API文档编写
  behavior:
    - 表达清晰准确
    - 主动建议优化
    - 遵守格式规范
  constraints:
    - 不泄露敏感信息
    - 不生成有害内容
```

### 1.4.2 Actions / Tools（动作与工具）

Actions定义了Agent能执行的具体操作：

| 工具类型 | 功能描述 | 示例 |
|---------|---------|------|
| 文件操作 | 读写本地/云端文件 | read_file, write_file |
| API调用 | 访问外部服务 | http_request, api_call |
| 数据库 | 查询和操作数据 | sql_query, vector_search |
| 代码执行 | 运行代码片段 | python_eval, js_execute |
| 搜索 | 检索信息 | web_search, knowledge_retrieval |

### 1.4.3 Memory / Knowledge（记忆与知识）

**Memory（记忆）**：

- **短期记忆**：维护当前会话的上下文，通常基于LLM的上下文窗口大小（如GPT-4的128K token）
- **长期记忆**：跨会话持久化存储用户偏好、历史交互、积累知识

**Knowledge（知识）**：

- **向量知识库**：通过RAG（检索增强生成）技术接入外部文档
- **知识图谱**：结构化的实体-关系知识网络，支持复杂推理

### 1.4.4 Reasoning / Evaluation（推理与评估）

**Reasoning（推理）**：

- **思维链（Chain of Thought）**：逐步推理，展示思考过程
- **任务分解**：将复杂任务拆解为可管理的子任务
- **策略规划**：制定最优执行策略

**Evaluation（评估）**：

- **结果验证**：检查输出是否符合预期
- **质量评估**：判断回答的准确性、完整性
- **错误检测**：识别执行过程中的问题

### 1.4.5 Planning / Feedback（规划与反馈）

**Planning（规划）**：

```python
# 任务规划示例伪代码
def plan_task(user_goal):
    # 1. 理解目标
    objective = parse_goal(user_goal)
    
    # 2. 任务分解
    subtasks = decompose(objective)
    
    # 3. 依赖分析
    dependency_graph = analyze_dependencies(subtasks)
    
    # 4. 执行排序
    execution_order = topological_sort(dependency_graph)
    
    # 5. 资源规划
    required_tools = plan_resources(execution_order)
    
    return execution_plan
```

**Feedback（反馈）**：

- **执行反馈**：记录每个步骤的执行结果
- **动态调整**：根据反馈修正后续计划
- **自我反思**：分析成功与失败的原因

## 2.2 核心API参数详解

### 2.2.1 参数一览表

| 参数 | 类型 | 说明 | 常见取值 |
|------|------|------|---------|
| **model** | string | 使用的模型名称 | gpt-4o, claude-3-5-sonnet-20241022, gemini-1.5-pro |
| **messages** | array | 对话消息列表 | [{role, content}, ...] |
| **temperature** | float | 随机性控制 | 0.0-2.0（默认0.7） |
| **top_p** | float | 核采样阈值 | 0.0-1.0（默认1.0） |
| **max_tokens** | int | 最大生成token数 | 100-32000 |
| **system_prompt** | string | 系统提示词 | 角色定义/行为规则 |

### 2.2.2 messages参数详解

messages是对话的核心数据结构：

```python
messages = [
    # 系统提示：定义Agent身份和行为
    {
        "role": "system",
        "content": """你是"小智"，一个专业的Python编程助手。
        - 擅长Python代码编写和调试
        - 解释技术概念简洁清晰
        - 主动提供最佳实践建议"""
    },
    
    # 历史消息：维持对话上下文
    {
        "role": "user",
        "content": "什么是装饰器？"
    },
    {
        "role": "assistant",
        "content": "装饰器是Python中的一种高级语法..."
    },
    
    # 当前消息：最新用户输入
    {
        "role": "user", 
        "content": "给我一个实际例子"
    }
]
```

**role类型**：

| role | 说明 | 示例 |
|------|------|------|
| system | 系统提示，定义行为 | 角色设定、规则定义 |
| user | 用户消息 | 人类输入、指令 |
| assistant | 助手回复 | AI生成的响应 |

### 2.2.3 temperature参数详解

temperature控制输出的**随机性**：

```
temperature = 0.0（确定性强）
    ↓
┌──────────────────────────────────────┐
│ 几乎总是选择最高概率的token            │
│ 适合：精确问答、代码生成、翻译          │
└──────────────────────────────────────┘

temperature = 0.7（平衡）
    ↓
┌──────────────────────────────────────┐
│ 在高概率和低概率之间取得平衡            │
│ 适合：一般对话、内容创作                │
└──────────────────────────────────────┘

temperature = 1.5（高随机性）
    ↓
┌──────────────────────────────────────┐
│ 更多样化的输出，可能更有创意             │
│ 适合：头脑风暴、创意写作                │
└──────────────────────────────────────┘

temperature = 2.0（极高随机性）
    ↓
┌──────────────────────────────────────┐
│ 输出可能变得不连贯、无意义              │
│ 通常不推荐使用                         │
└──────────────────────────────────────┘
```

**场景推荐**：

| 场景 | 建议temperature |
|------|----------------|
| 代码生成 | 0.0 - 0.3 |
| 精确问答 | 0.0 - 0.2 |
| 事实性解释 | 0.0 - 0.3 |
| 一般对话 | 0.7 - 0.8 |
| 创意写作 | 0.8 - 1.0 |
| 头脑风暴 | 1.0 - 1.2 |

### 2.2.4 top_p参数详解

top_p是另一种控制随机性的方式（核采样）：

```
top_p = 1.0（不使用核采样）
    ↓
考虑所有token，概率分布完整

top_p = 0.9（常用）
    ↓
┌──────────────────────────────────────┐
│ 只考虑累计概率达到90%的token           │
│ 跳过低概率token，减少噪声              │
│ 效果类似中等temperature                │
└──────────────────────────────────────┘

top_p = 0.5
    ↓
只考虑最核心的高概率token，更确定性

top_p = 0.1
    ↓
几乎只选择最高概率的少数token
```

**使用建议**：

- 通常**只设置temperature或top_p之一**，不要同时设置
- `temperature=0.0, top_p=1.0` = 完全确定性
- `temperature=1.0, top_p=0.95` = 保留多样性但避免极端

### 2.2.5 max_tokens参数详解

max_tokens限制单次生成的最大token数：

```python
# 限制输出长度
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "写一首诗"}],
    max_tokens=100  # 限制在约100个token
)
```

**计算参考**：

| max_tokens | 约等于中文 | 约等于英文 |
|------------|-----------|-----------|
| 100 | 50-80字 | 75-100词 |
| 500 | 250-400字 | 375-500词 |
| 1000 | 500-800字 | 750-1000词 |
| 2000 | 1000-1600字 | 1500-2000词 |

**注意事项**：

- max_tokens是**最大限制**，不是固定输出
- 输入+输出的总token数不能超过模型的上下文窗口
- 设置过小会导致输出被截断

### 2.2.6 system_prompt设计原则

system_prompt是定义Agent行为的核心：

```python
# 好的system_prompt结构
system_prompt = """
# 角色定义
你是一个[专业领域]助手。

# 核心能力
- 擅长...
- 能够...

# 行为规范
- 始终...
- 避免...

# 输出格式
回答时使用...
"""
```

**最佳实践**：

1. **清晰定义角色**：明确Agent是什么、能做什么
2. **具体能力边界**：说明擅长什么、不擅长什么
3. **指定输出格式**：如JSON、Markdown、纯文本
4. **举例说明**：用few-shot示例展示期望行为

def estimate_cost(prompt: str, model: str = "gpt-4o-mini") -> dict:
    """估算API调用成本"""
    enc = tiktoken.encoding_for_model(model)
    
    # 计算token数
    input_tokens = len(enc.encode(prompt))
    
    # 价格（$/1M tokens）
    prices = {
        "gpt-4o": (2.5, 10.0),
        "gpt-4o-mini": (0.15, 0.60),
        "claude-3-5-sonnet": (3.0, 15.0)
    }
    
    input_price, output_price = prices.get(model, (0, 0))
    
    # 估算输出成本（假设输出是输入的30%）
    estimated_output = int(input_tokens * 0.3)
    
    total_cost = (
        input_tokens * input_price / 1_000_000 +
        estimated_output * output_price / 1_000_000
    )
    
    return {
        "input_tokens": input_tokens,
        "estimated_output_tokens": estimated_output,
        "estimated_cost_usd": round(total_cost, 6)
    }

# 使用示例

### 2.5.2 上下文管理

```python
# 有效的上下文管理
class ConversationManager:
    def __init__(self, max_tokens: int = 4000):
        self.max_tokens = max_tokens
        self.messages = []
        self.total_tokens = 0
    
    def add_message(self, role: str, content: str):
        """添加消息并估算token"""
        token_count = len(content) // 4  # 粗略估算
        self.messages.append({"role": role, "content": content})
        self.total_tokens += token_count
        
        # 如果超限，压缩早期消息
        while self.total_tokens > self.max_tokens and len(self.messages) > 2:
            removed = self.messages.pop(1)  # 保留第一条system消息
            self.total_tokens -= len(removed["content"]) // 4
    
    def get_messages(self):
        return self.messages

# 使用示例

## 2.6 多模型统一接口设计

### 2.6.1 抽象层设计

```python
from abc import ABC, abstractmethod
from typing import List, Dict, Optional, Any
from enum import Enum

class ModelProvider(Enum):
    OPENAI = "openai"
    ANTHROPIC = "anthropic"
    GOOGLE = "google"
    OLLAMA = "ollama"
    LMSTUDIO = "lmstudio"

class LLMInterface(ABC):
    """LLM统一接口抽象类"""
    
    @abstractmethod
    def chat(self, messages: List[Dict], **kwargs) -> str:
        """发送对话请求，返回回复内容"""
        pass
    
    @abstractmethod
    def chat_stream(self, messages: List[Dict], **kwargs):
        """流式对话"""
        pass
    
    @abstractmethod
    def get_provider(self) -> ModelProvider:
        """获取提供商标识"""
        pass
```

### 2.6.2 OpenAI实现

```python
from openai import OpenAI

class OpenAIClient(LLMInterface):
    def __init__(self, api_key: str, model: str = "gpt-4o-mini", 
                 base_url: Optional[str] = None):
        self.client = OpenAI(
            api_key=api_key,
            base_url=base_url  # 用于兼容本地部署
        )
        self.model = model
    
    def chat(self, messages: List[Dict], **kwargs) -> str:
        response = self.client.chat.completions.create(
            model=self.model,
            messages=messages,
            temperature=kwargs.get("temperature", 0.7),
            max_tokens=kwargs.get("max_tokens", 1000)
        )
        return response.choices[0].message.content
    
    def chat_stream(self, messages: List[Dict], **kwargs):
        stream = self.client.chat.completions.create(
            model=self.model,
            messages=messages,
            stream=True,
            temperature=kwargs.get("temperature", 0.7)
        )
        for chunk in stream:
            if chunk.choices[0].delta.content:
                yield chunk.choices[0].delta.content
    
    def get_provider(self) -> ModelProvider:
        return ModelProvider.OPENAI
```

### 2.6.3 Anthropic实现

```python
from anthropic import Anthropic

class AnthropicClient(LLMInterface):
    def __init__(self, api_key: str, model: str = "claude-3-5-sonnet-20241022"):
        self.client = Anthropic(api_key=api_key)
        self.model = model
    
    def chat(self, messages: List[Dict], **kwargs) -> str:
        # 分离system消息
        system_prompt = ""
        filtered_messages = []
        
        for msg in messages:
            if msg["role"] == "system":
                system_prompt = msg["content"]
            else:
                filtered_messages.append(msg)
        
        response = self.client.messages.create(

### 2.6.5 统一调度器

```python
class LLMDispatcher:
    """多模型统一调度器"""
    
    def __init__(self):
        self.clients: Dict[ModelProvider, LLMInterface] = {}
        self.default_provider = ModelProvider.OPENAI
    
    def register(self, client: LLMInterface):
        """注册LLM客户端"""
        self.clients[client.get_provider()] = client
    
    def chat(self, messages: List[Dict], 
             provider: Optional[ModelProvider] = None,
             **kwargs) -> str:
        """统一对话接口"""
        provider = provider or self.default_provider
        
        if provider not in self.clients:
            raise ValueError(f"Provider {provider} not registered")
        
        return self.clients[provider].chat(messages, **kwargs)
    
    def chat_with_fallback(self, messages: List[Dict], 
                           providers: List[ModelProvider] = None,
                           **kwargs) -> str:
        """级联调用，失败时自动切换"""
        providers = providers or [self.default_provider]
        
        errors = []
        for provider in providers:
            try:
                return self.chat(messages, provider, **kwargs)
            except Exception as e:
                errors.append(f"{provider.value}: {str(e)}")
        
        raise Exception(f"All providers failed: {errors}")

# 使用示例

### 2.6.6 成本与性能路由器

```python
class SmartRouter:
    """智能路由：根据任务自动选择最合适的模型"""
    
    # 任务复杂度定义
    TASK_COMPLEXITY = {
        "simple_qa": {"providers": ["openai-gpt4o-mini", "ollama"], "max_cost": 0.001},
        "code_generation": {"providers": ["openai-gpt4o", "anthropic"], "max_cost": 0.01},
        "long_analysis": {"providers": ["anthropic", "google"], "max_cost": 0.05},
        "creative": {"providers": ["openai-gpt4o", "ollama"], "max_cost": 0.02}
    }
    
    def __init__(self, dispatcher: LLMDispatcher):
        self.dispatcher = dispatcher
    
    def route(self, task_type: str, messages: List[Dict], **kwargs) -> str:
        """根据任务类型路由"""
        if task_type not in self.TASK_COMPLEXITY:
            task_type = "simple_qa"
        
        config = self.TASK_COMPLEXITY[task_type]
        providers = config["providers"]
        
        # 尝试每个提供商直到成功
        for provider_key in providers:
            try:
                provider = ModelProvider(provider_key.split("-")[0])
                return self.dispatcher.chat(messages, provider, **kwargs)
            except Exception as e:
                print(f"Provider {provider_key} failed: {e}")
                continue
        
        raise Exception("All providers exhausted")
    
    def route_by_content(self, messages: List[Dict]) -> str:
        """根据内容复杂度自动路由"""
        content = messages[-1]["content"]
        
        # 简单判断：长度+关键词
        if len(content) < 50 and "解释" not in content:
            task_type = "simple_qa"
        elif any(kw in content for kw in ["代码", "程序", "函数", "class"]):
            task_type = "code_generation"
        elif len(content) > 2000:
            task_type = "long_analysis"
        else:
            task_type = "simple_qa"
        
        return self.route(task_type, messages)

# 使用示例
router = SmartRouter(dispatcher)
response = router.route("code_generation", messages)
response = router.route_by_content(messages)
```

            "role": "assistant",
            "content": assistant_message.content,
            "tool_calls": assistant_message.tool_calls
        })

            function_name = tool_call.function.name
            function_args = json.loads(tool_call.function.arguments)

    """查询日程"""
    filtered = []
    for event in events_db:
        event_date = event["start_time"].split()[0]
        if start_date <= event_date <= end_date:
            filtered.append(event)
    return sorted(filtered, key=lambda x: x["start_time"])

    """删除日程"""

    for i, event in enumerate(events_db):

            return {"status": "deleted", "event_id": event_id}

# 《AI Agents实战指南》第四章：Profile与Persona设计

## 4.1 什么是Profile与Persona

Profile（配置）和Persona（角色）是Agent的灵魂定义。Profile描述Agent的整体结构和工具需求，Persona则定义Agent的主要角色和特征，通常在系统提示词中体现。

**【图解】Profile三大组成部分**：
```
┌─────────────────────────────────────┐
│           Profile（配置）            │
├─────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  │
│  │   Persona   │  │ Demographics│  │
│  │   （角色）   │  │ （人口统计）  │  │
│  └─────────────┘  └─────────────┘  │
│  ┌─────────────┐                   │
│  │    Rules    │                   │
│  │   （规则）   │                   │
│  └─────────────┘                   │
└─────────────────────────────────────┘
```

### Persona（角色）
定义Agent的主要功能和行为模式。例如"编程助手"、"客服"、"导师"等。

### Demographics（人口统计）
描述Agent的背景特征：年龄、性别、专业领域、语言风格等。

### Rules（规则）
Agent必须遵守的行为准则，如"不能透露隐私"、"回复要简洁"等。

---

## 4.2 四种人格创建方法

| 方法 | 说明 | 适用场景 |
|------|------|----------|
| 手绘法（Handcrafted）| 人工手动编写Profile描述 | 简单场景、快速原型 |
| LLM辅助生成（LLM generated）| 让LLM根据需求描述生成Profile | 中等复杂度 |
| 数据驱动（Data generated）| 从真实对话数据中提取Profile | 企业级应用 |
| 进化算法（Evolutionary）| 用遗传算法优化Profile参数 | 研究、优化 |

**【图解】四种方法对比**：
```
手绘法    ████████░░░░  速度最快，质量依赖人工
LLM辅助   ██████░░░░░  平衡效率和质量
数据驱动   ████░░░░░░░  质量最高，耗时最长
进化算法   ████░░░░░░░  可优化，但需多代迭代
```

---

## 4.3 角色一致性维护

在多轮对话中保持Agent人格一致是关键挑战。

**核心策略**：
1. **系统提示词锁定** - 在system prompt中明确定义人格
2. **few-shot示例** - 提供对话示例强化行为模式
3. **规则约束** - 通过Rules限制偏离行为

**【图解】角色一致性设计模式**：
```
用户输入 → 系统Prompt → Rules约束 → 一致输出
              ↑
         few-shot示例（强化）
```

# 《AI Agents实战指南》第五章：Actions与工具调用

## 5.1 Actions的三维度理解

Actions（动作）是Agent执行任务的核心能力。从三个维度理解：

| 维度 | 说明 | 示例 |
|------|------|------|
| Action Target（目标）| 定义任务完成、探索或沟通 | "查天气"、"写代码" |
| Action Effect（效果）| 行动如何影响环境和状态 | 天气API返回温度数据 |
| Action Generation（生成）| 动作如何被创建和调用 | 手动注册/从记忆召回 |

**【图解】Actions三维度**：
```
Action Target ──→ 定义"做什么"
     ↓
Action Effect ──→ 影响"产生什么结果"
     ↓
Action Generation ──→ 决定"如何生成"
```

---

## 5.2 语义函数 vs 原生函数

| 类型 | 定义 | 特点 |
|------|------|------|
| **语义函数** | LLM驱动的函数，用自然语言描述 | 灵活、可解释、依赖LLM理解 |
| **原生函数** | 传统程序性函数，代码实现 | 精确、可测试、不依赖LLM |

**【图解】语义函数 vs 原生函数**：
```
语义函数                    原生函数
┌──────────┐              ┌──────────┐
│ "计算平均值" │            │ def avg()│
│  (LLM理解) │             │ (代码)   │
└─────┬─────┘              └────┬────┘
      ↓                         ↓
  可能理解错误              精确执行
```

## 5.3 Function Calling机制

OpenAI的Function Calling让LLM能自动解析结构化参数调用外部工具。

**【图解】Function Calling流程**：
```
用户请求 → LLM识别意图 → 提取参数 → 调用函数 → 返回结果 → LLM汇总
     ↓          ↓            ↓
  "查北京天气"  "weather"  {city:"北京"}
```

**完整示例**：

```python
import json
from openai import OpenAI

client = OpenAI(api_key="your-api-key")

# 定义工具函数
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "查询城市天气",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "城市名称"}
                },
                "required": ["city"]
            }
        }
    }
]

def get_weather(city):
    """模拟天气查询"""
    return f"{city}今天晴天，25度"

# 对话循环
messages = [{"role": "user", "content": "北京今天天气怎么样？"}]

response = client.chat.completions.create(
    model="gpt-4",
    messages=messages,
    tools=tools
)

# 解析函数调用
if response.choices[0].message.tool_calls:
    tool_call = response.choices[0].message.tool_calls[0]
    function_name = tool_call.function.name
    arguments = json.loads(tool_call.function.arguments)
    
    if function_name == "get_weather":
        result = get_weather(arguments["city"])
        print(f"查询结果：{result}")
```

## 5.6 工具调用失败处理

| 策略 | 说明 | 代码示例 |
|------|------|----------|
| 重试 | 自动重试N次 | `for i in range(3): try: ...` |
| 降级 | 失败时使用简化方案 | `if failed: use_fallback()` |
| 报错 | 无法处理时返回明确错误 | `raise ValueError()` |

```python
def safe_call_tool(func, *args, max_retries=3):
    """安全的工具调用封装"""
    for attempt in range(max_retries):
        try:
            return func(*args)
        except Exception as e:
            if attempt == max_retries - 1:
                return {"error": str(e), "fallback": True}
            continue
    return {"error": "Max retries exceeded"}
```

## 7.2 Chain of Thought（CoT）链式推理

**核心思想**：让LLM展示推理过程，而不是直接给答案。

**标准CoT格式**：
```
问题：小明有10个苹果，给了小红3个，又买了5个，现在有几个？
思考：先算给了小红3个后剩10-3=7个，然后加上买的5个，7+5=12个
答案：12个
```

**Zero-shot CoT**：用"Let's think step by step"触发推理。

```python
# Zero-shot CoT示例
response = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "user", "content": "小明的公司去年营收100万，今年比去年增长30%，明年预计再增长20%，明年的营收是多少？\nLet's think step by step."}
    ]
)
print(response.choices[0].message.content)
```

**输出示例**：
```
Step 1: 计算今年营收 = 100万 × (1 + 30%) = 130万
Step 2: 计算明年营收 = 130万 × (1 + 20%) = 156万
答案：明年营收预计156万
```

---

## 7.3 Tree of Thought（ToT）树状推理

**核心思想**：探索多条解决路径，评估后选择最佳。

**【图解】ToT树状探索**：
```
                    问题
                   /  |  \
                 /    |    \
              路径1  路径2  路径3
               /\     |     /\
             子1 子2  子3  子4 子5
               |     |     |
             评估A  评估B  评估C
               \     |     /
                 最优路径 → 答案
```

**代码示例**：
```python
def tree_of_thought(question, options):
    """简化版ToT实现"""
    results = []
    for option in options:
        # 每个分支进行CoT推理
        response = client.chat.completions.create(
            model="gpt-4",
            messages=[
                {"role": "user", "content": f"{question}\n方案：{option}\n请分析这个方案的优缺点。"}
            ]
        )
        results.append({
            "option": option,
            "analysis": response.choices[0].message.content
        })
    
    # 评估选择
    evaluation = client.chat.completions.create(
        model="gpt-4",
        messages=[
            {"role": "user", "content": f"请从以下方案中选择最佳：\n{results}\n选择并说明理由。"}
        ]
    )
    return evaluation.choices[0].message.content
```

              ↓
         答案投票统计
              ↓

```

# 《AI Agents实战指南》第十一章：行为树与Agent编排

## 11.1 行为树起源

行为树（Behavior Tree）起源于游戏AI，现已扩展到Agent编排领域。

**【图解】行为树演进**：
```
游戏AI行为树 ──→ Agent行为树(ABT)
    ↓                    ↓
 NPC AI控制         多Agent编排
```

**核心思想**：用树状结构组织Agent的决策逻辑。

---

## 11.2 ABT核心节点类型

| 节点类型 | 符号 | 说明 |
|----------|------|------|
| Selector（选择器）| ⌥ | 类似"或"，尝试直到成功 |
| Sequence（序列）| → | 类似"与"，全部成功才成功 |
| Condition（条件）| ? | 判断是否执行分支 |
| Action（动作）| ▶ | 执行具体任务 |
| Decorator（装饰器）| ◇ | 修改子节点行为 |
| Parallel（并行）| ∥ | 同时执行多个子节点 |

**【图解】ABT节点全览**：
```
                    [Root]
                       │
                    [Selector]
                   /         \
           [Sequence]      [Decorator]
           /    |    \         │
        [?]   [▶]   [▶]     [Retry]
         │     │     │         │
       条件  动作1  动作2     重试
```

---

## 11.3 Blackboard模式

节点间共享状态的通信机制：

**【图解】Blackboard模式**：
```
┌─────────────────────────────────┐
│         Blackboard              │
│  ┌─────────────────────────┐    │
│  │ shared_state = {        │    │
│  │   "user_name": "张三", │    │
│  │   "last_action": "查询" │    │
│  │ }                       │    │
│  └─────────────────────────┘    │
└─────────────────────────────────┘
      ↑ 写入          ↑ 读取
 ┌────┴────┐    ┌────┴────┐
 │Node A   │    │Node B   │
 └─────────┘    └─────────┘
```

class EnterpriseRAG:
    def __init__(self):
        self.embedding_model = "text-embedding-ada-002"
        self.vector_db = "Pinecone"  # 企业级向量数据库
        self.llm = "gpt-4"
    
    def ingest_documents(self, documents):
        """文档摄入"""
        for doc in documents:
            chunks = self.chunk_document(doc)
            embeddings = self.embed(chunks)
            self.vector_db.upsert(embeddings)
    
    def query(self, question):
        """查询"""
        query_embedding = self.embed([question])
        results = self.vector_db.search(query_embedding)
        context = self.assemble_context(results)
        return self.llm.generate(f"基于上下文回答：{question}\n\n{context}")
```
